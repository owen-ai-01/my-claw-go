# 当前 Runtime 容量估算（available 内存约 3G）2026-04-21

> 日期：2026-04-21
> 场景：服务器当前 `free -m` 里可用内存大约还有 `3G`，需要估算这台机器还能支持几个订阅用户创建容器，以及能稳定支撑几个用户同时在线使用。

---

## 1. 核心结论

如果你说的是 `free -m` 里的 **available 大约 3G**，那按你当前代码里的真实容器内存限制来看：

`安全能稳定支撑的并发，大致只有 1 个 Pro 用户。`

更明确地说：

- 可以创建很多“未启动”的容器
- 可以挂起不止 1 个空闲容器
- 但如果按“稳定同时在线使用”来算，现在这台机子只适合按 `1 个 Pro 并发活跃用户` 理解

`Premium` 和 `Ultra` 目前都不应该按“可稳定承载”来卖。

---

## 2. 这个结论是按什么算的

你当前代码里，runtime 容器的内存限制写在：

- `src/lib/myclawgo/docker-manager.ts`

实际配置是：

- Pro: `2g`
- Premium: `4g`
- Ultra: `8g`

而且创建容器时还设置了：

- `--memory = 套餐内存`
- `--memory-swap = 套餐内存`

这意味着：

- 容器内存限制是硬上限
- 没有额外 swap 缓冲
- 一旦内存紧张，压力会更直接

所以不能把“有 3G available”简单理解成“还能轻松开多个订阅容器”。

---

## 3. 必须区分的 3 种情况

### 3.1 可以创建几个容器

如果只是“创建容器”但不启动，那么数量可以很多。

原因是：

- 未运行的容器几乎不占运行内存
- 磁盘和元数据才是主要消耗

所以从“创建几个容器”的角度看，这个数并不关键。

### 3.2 可以同时启动几个空闲容器

如果容器只是启动着，但基本不工作，理论上可能能挂不止 1 个 Pro 容器。

因为：

- Docker 的 memory limit 不是一启动就立刻把 2G 全占满
- 空闲容器的实际 RSS 可能远低于 limit

但这不代表它们在用户真正使用时还能稳。

### 3.3 可以稳定支撑几个在线用户

这是最重要的指标。

如果按“用户真的在用，容器会开始吃内存”的标准算，你不能把 available 3G 全部拿来给容器。

因为还要给这些东西留余量：

- 宿主机系统
- Docker daemon
- Nginx
- PM2
- Next.js 主站
- 其他后台进程

所以 available 3G 并不等于“容器可用 3G”。

---

## 4. 按当前情况的保守估算

一个更实用的公式是：

```text
安全并发数 ≈ floor((available内存 - 0.5~1G系统余量) / 套餐内存上限)
```

如果按你现在 `available ≈ 3G` 来估：

### Pro

- 容器限制：`2G`
- 预留宿主机余量：`0.5 ~ 1G`

那么安全估算大概是：

- `(3 - 1) / 2 = 1`

也就是：

- **安全并发：1 个 Pro**

### Premium

- 容器限制：`4G`

那么现在：

- `(3 - 1) / 4 = 0`

也就是：

- **不适合稳定承载 Premium 并发**

### Ultra

- 容器限制：`8G`

那么现在：

- `(3 - 1) / 8 = 0`

也就是：

- **不适合稳定承载 Ultra 并发**

---

## 5. 更直接的现实判断

如果按你当前机器状态来讲，更接近真实情况的说法是：

- 还可以继续给新用户创建 runtime 容器记录
- 也可能还能启动一些空闲容器
- 但如果要按“用户一上来就真实聊天、真实跑任务”的标准卖订阅
- 当前机器只适合按 `1 个 Pro 用户稳定在线` 来判断

这不是因为代码写错，而是因为：

- Pro 本身就是 `2G`
- 你当前 available 只有 `3G`
- 宿主机和主站也要吃内存

所以没有足够缓冲去同时稳定跑多个活跃容器。

---

## 6. 为什么这里不能只看 free 内存

还有一个很重要的点：

你提到的是 `free -m`。

这里真正应该重点看的是：

- `available`

而不是：

- `free`

因为 Linux 的 `free` 可能很低，但 cache 可以回收。  
真正更接近“还能不能安全再承载负载”的是 `available`。

所以如果你说的 `3G` 其实是：

- `free` 列，不是 `available` 列

那这个估算就还偏乐观，实际需要重新按 `available` 来算。

---

## 7. 对你当前售卖策略的建议

结合当前内存情况，建议这样理解：

### 当前机器适合什么

- 继续验证产品
- 少量 Pro 用户试用或低并发使用
- 容器“按需启动”，不是多人同时跑

### 当前机器不适合什么

- 稳定承载多个 Pro 用户同时在线
- 承诺 Premium 可稳定可用
- 承诺 Ultra 可稳定可用

### 更保守的业务判断

如果你现在真的只有 `available 约 3G`，那最稳的做法是：

- 只把当前机器视作“还能再接 1 个 Pro 活跃用户”
- 一旦出现第一个真实持续在线的付费用户，就准备升配或扩容

---

## 8. 最终结论

对你这台机器，在 `available 内存约 3G` 的前提下：

- **可创建容器数量**：很多，重点不在这里
- **可启动空闲容器数量**：可能不止 1 个，但不代表稳
- **可稳定承载订阅用户并发**：大致只有 `1 个 Pro`

所以最直接的说法是：

`现在这台机器还能继续跑，但如果按稳定订阅服务来卖，最好只按 1 个 Pro 活跃用户来理解。`
